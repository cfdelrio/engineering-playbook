# 01 — Producto

> La ingeniería sirve al producto. El producto sirve a un usuario con un problema real.

## El portafolio

Este playbook gobierna varios productos. Comparten doctrina, no dominio.

| Producto | En una línea |
|---|---|
| **AURORA** | Plataforma de entrenamiento y análisis de rendimiento deportivo. |
| **ProdeCaballito** | Juego de pronósticos / quiniela social. |
| **Engage** | Producto de engagement / relación con audiencias. |
| **Storysell** | Venta y narrativa de producto (commerce + storytelling). |

> Mantené esta tabla viva. Cada producto debe tener su propio `MANIFESTO.md` con su visión específica.

## Cómo pensamos un producto

1. **Problema antes que solución.** ¿Qué duele, a quién, y cuánto?
2. **Usuario antes que feature.** Una feature sin usuario claro es una opinión.
3. **Resultado antes que output.** Entregamos cambios en la vida del usuario, no líneas de código.
4. **Lo mínimo que enseña.** El primer corte existe para aprender, no para impresionar.

## De producto a ingeniería

Toda iniciativa de producto se traduce a ingeniería mediante:

1. Una **visión de producto** (`product-vision.md` en el proyecto).
2. Un **modelo de dominio** que captura los conceptos del negocio.
3. Una o más **feature specs** que definen comportamiento concreto.
4. Las **decisiones de arquitectura** (ADRs) que habilitan esas features.

Si no podés trazar una línea desde "esto le sirve al usuario" hasta "por eso escribo este código", parate y revisá.

## Métricas que importan

Cada producto define sus métricas norte. Por defecto preferimos pocas y accionables sobre muchas y decorativas. Lo que se mide, se mejora; lo que se mide mal, se distorsiona.
